# EIP-7938 / EIP-7999 / EIP-8200 crosswalk

Scope: official EIP specs and their discussion threads, with a focus on bottlenecks, mechanism, dependencies, backwards compatibility, objections, and likely relevance to Strawmap items.

## EIP-7938: Exponential Gas Limit Increase

### Bottleneck
The bottleneck is not a protocol execution limit; it is coordination. The proposal says the current gas-limit voting mechanism can stagnate or remain overly cautious because it depends on miner/operator voting. It tries to replace ad hoc coordination with a predictable schedule. The discussion thread shows the main pushback: concern that the schedule is too fast, that it should be time-based rather than epoch-based, and that increasing gas limit optimistically could outrun state-growth and client readiness.

### Mechanism
The spec sets a default client voting schedule: start from `G0 = 50,000,000` at activation epoch `369017`, increase exponentially every epoch according to a 2-year, 10x schedule, and cap at `100 * G0` after 4 years unless revised. Clients continue using the existing gas-voting mechanism, but the default vote changes. Users can override it.

### Dependencies
This is intentionally non-consensus and depends on existing client gas-voting behavior rather than a fork-level rule change. It implicitly depends on the beacon-chain epoch cadence and on all major clients deciding to adopt the default vote behavior.

### Backwards compatibility
The EIP is explicitly non-consensus and backward compatible. Clients that do not implement it keep their current behavior; only the default changes.

### Objections / counter-search
Primary objections from the discussion thread:
- The schedule may be too fast and could front-run demand.
- Epoch-count-based scheduling could accelerate unintentionally if epoch length changes.
- Gas-limit growth should wait for demonstrated client capacity, not assume future performance improvements.
- State growth could become unsustainable at 100x.

### Strawmap relevance
This is directly relevant to `gas limit increases` and `gigagas L1`. It is the clearest of the three for scaling block capacity, but it is also the most politically and operationally sensitive because it shifts the coordination burden to client defaults and raises state-growth concerns.

## EIP-7999: Unified multidimensional fee market

### Bottleneck
The bottleneck is fee-market fragmentation across resources. The EIP says directly that users currently need separate fee caps per resource, which is bad for UX and capital efficiency. It also identifies technical debt from having one mechanism for EVM gas and another for blob gas.

### Mechanism
The proposal introduces:
- one aggregate `max_fee` budget for all resources,
- unified fee accounting across dimensions,
- a new multidimensional transaction type,
- a vectorized block gas model,
- calldata as the first added resource,
- reserve pricing generalized from EIP-7918,
- normalization so price behavior stays stable when limits change.

The interesting design choice is that `max_fee` becomes fungible across resources. The protocol checks whether the aggregate budget can cover the required fees across all dimensions, rather than requiring a separate budget for each one.

### Dependencies
The spec explicitly requires EIP-1559, EIP-2718, EIP-4844, EIP-7516, EIP-7691, EIP-7840, and EIP-7918. The rationale also leans on EIP-7706 and EIP-7805 as precursor ideas for multidimensional gas and censorship resistance.

### Backwards compatibility
The design is meant to preserve compatibility via a new transaction type and via explicit handling for older transaction formats. The rationale also discusses aggregate-gas and hybrid models to retain gas observability for legacy contracts, but it admits that old contracts can still break because repricing can change behavior and gas-capped subcalls are sensitive.

### Objections / counter-search
The discussion thread raises several hard questions:
- In a metering-only or max-based scheme, can builders bias packing so basefee stays artificially low while priority fees remain high?
- What evidence or simulation would show that adversarial packing is not a practical problem?
- How should the implementation avoid growing complexity while still handling more dimensions?

The EIP itself also acknowledges concern about UX and complexity, and it explicitly notes that expanding to more dimensions gets harder, especially once EVM gas itself is split into multiple dimensions.

### Strawmap relevance
This is directly relevant to `multidimensional pricing`. It is also relevant to `gigagas L1` if Strawmap wants a scaling path that is not just “raise gas limit,” because 7999 provides the fee-market machinery needed to scale multiple constrained resources at once.

It is less directly about EVMifying long-tail precompiles, but it is structurally important if those precompiles become one of several priced dimensions or if Strawmap wants fee abstraction for different EVM resource classes.

## EIP-8200: EVMification

### Bottleneck
The bottleneck is precompile special-casing. The EIP argues that precompiles create ongoing maintenance cost, increase consensus-bug risk across clients, and make zkEVM implementations harder because each precompile needs a separate optimized non-EVM implementation.

### Mechanism
The proposal replaces selected precompiles with EVM bytecode deployed at the same addresses at fork activation. That preserves the call interface while shifting the implementation into ordinary EVM code.

### Dependencies
The EIP requires EIP-152, EIP-2537, EIP-7666, EIP-7823, and EIP-7883. It also inherits the deployment pattern from EIP-7666, which already EVMifies the identity precompile.

### Backwards compatibility
The spec’s main compatibility claim is address preservation: existing contracts keep calling the same addresses, which now hold code instead of being treated as native precompiles. The EIP admits that gas costs will change, and contracts that depend on exact gas formulas may break.

### Objections / counter-search
The discussion thread’s main objection is performance:
- precompiles are fast,
- native execution is often much cheaper than equivalent EVM bytecode,
- the proposal needs strong justification if it risks turning a cheap native primitive into something slower or more expensive.

There is also a stronger concern in the thread that gas costs should reflect computational complexity, not just simplicity of implementation.

### Strawmap relevance
This is directly relevant to `EVMify long-tail precompiles`. It is also relevant to any Strawmap plan that wants to reduce client-special-case burden while preserving the external API shape of old primitives. For `gigagas L1`, it matters insofar as removing precompile special cases may simplify client engineering at higher throughput levels, but it may also move work into the EVM and increase execution cost.

## Crosswalk

### Mechanism fit
- `gas limit increases`: best matched by EIP-7938.
- `multidimensional pricing`: best matched by EIP-7999.
- `EVMify long-tail precompiles`: best matched by EIP-8200.
- `gigagas L1`: depends on the combination of 7938 for capacity growth and 7999 for resource-specific pricing; 8200 is an enabler only if precompile special-casing becomes an operational bottleneck at scale.

### Dependency fit
- 7938 is almost standalone and can be treated as a client-default policy change.
- 7999 is deeply dependent on prior fee-market and multidimensional-gas work.
- 8200 depends on prior EVMification work and precompile-specific prerequisite EIPs.

### Likely ordering for Strawmap
1. If Strawmap needs quick capacity expansion without redesigning fee logic, 7938 is the shortest path.
2. If Strawmap needs a principled price model for multiple constrained resources, 7999 is the architecture piece.
3. If Strawmap needs to reduce client maintenance and eliminate long-tail special cases, 8200 is the cleanup/refactoring piece.

## CLAIMS

1. EIP-7938 makes gas-limit growth a deterministic client default, not a consensus rule.
2. EIP-7938 is non-consensus and backward compatible.
3. EIP-7938’s primary objections are schedule speed, epoch-cadence sensitivity, and state-growth risk.
4. EIP-7999 requires EIP-1559, EIP-2718, EIP-4844, EIP-7516, EIP-7691, EIP-7840, and EIP-7918.
5. EIP-7999’s main mechanism is a single aggregate `max_fee` across resources.
6. EIP-7999 is explicitly designed to preserve compatibility through a new transaction type and aggregate/hybrid gas paths, but it still admits repricing-related breakage risk.
7. EIP-7999’s main objections are adversarial packing, simulation/complexity concerns, and fee-market pathologies under multidimensional limits.
8. EIP-8200 replaces specific precompiles with EVM bytecode at the same addresses.
9. EIP-8200’s main compatibility guarantee is address preservation, but it changes gas costs and can break gas-sensitive callers.
10. EIP-8200’s main objection is that precompiles are faster and simpler to justify than equivalent EVM code unless the maintenance/zkEVM argument outweighs the performance loss.

## EXPAND

Open research questions for the next wave:
- Which Strawmap target resource is most constrained today: execution gas, calldata, blob-like data, or client complexity?
- Does Strawmap want a near-term capacity hack (7938) or a structural fee-market redesign (7999) first?
- Which precompiles are actually long-tail enough that evmification is net-positive under realistic usage and gas-cost deltas?
- If Strawmap wants gigagas L1, what is the minimum fee-market and client-engineering stack needed before that becomes safe?
